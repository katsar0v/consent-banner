Overview

Consent Banner renders a configurable consent interface in WordPress, keeps essential cookies always enabled, records consent decisions, and exposes APIs for themes, plugins, and deployment workflows.

Consent choices

Visitors can accept all, reject all, or customize consent at category level.

Required essential category

The essential category is enforced as required, while custom categories can be managed in admin.

Integration surface

Use REST routes, a JavaScript API, a PHP helper, shortcodes, hooks, filters, and WP-CLI import/export.

Script auto-blocking is intentionally out of scope for version 0.2.0. This release records consent and exposes APIs/hooks so integrations can decide how to react.

Key Features

The plugin focuses on clear visitor consent flows, manageable admin settings, deployable configuration, and practical integration points.

Consent UX

Accept all, reject all, and per-category customization inside a modal preferences flow.

Category model

Essential cookies stay enabled by design; categories are adjustable and localized.

Bilingual text

English and Bulgarian text packs are selected from the active site locale.

Import and export

Move JSON settings through the admin UI or WP-CLI, with optional version bumping.

REST endpoints

Public config and consent submission routes plus admin-only settings routes.

Audit option

Optional hashed consent logging can store proof entries without keeping raw visitor identifiers.

Admin Settings

The settings page lives under Settings -> Consent Banner and currently exposes three enabled tabs.

Tab Status Purpose
General settingsEnabledCategories, texts, lifetime, consent version, and uninstall behavior.
AppearanceEnabledBanner position, theme, and visual behavior.
Export / ImportEnabledDownload or restore JSON settings from the admin UI.

External scripts and TCF tabs exist in code but are disabled in version 0.2.0, so they should not be described as active user-facing features.

First-Time Setup

1

Open Settings -> Consent Banner in wp-admin.

2

Confirm categories and keep essential required.

3

Review English and Bulgarian texts for the banner and preferences modal.

4

Set banner position, visual theme, and consent lifetime.

5

Save settings. Optionally bump consent version when users must choose again.

Frontend Behavior

The frontend renderer mounts the banner in wp_footer and hides it when a valid consent cookie exists for the current consent version.

Accept all

Enables all configured categories.

Reject all

Enables only required categories.

Customize

Opens category toggles; essential is locked on.

REST API

The primary namespace is /wp-json/kdconsent/v1. Legacy compatibility also registers /wp-json/kdcb/v1 for older integrations.

Methods Endpoint Access Purpose
GET/configPublicRuntime config, texts, categories, behavior, and current consent state.
POST/consentPublic, rate-limitedSubmit and persist a consent decision.
GET/settingsAdmin onlyRead plugin settings.
PUT, PATCH/settingsAdmin onlyUpdate plugin settings.

Admin REST requests require X-WP-Nonce and the manage_options capability.

WP-CLI Import and Export

Use WP-CLI for deployment backups, environment sync, and dry-run validation of settings JSON.

Export

wp consent-banner export /tmp/consent-banner-settings.json
wp consent-banner export /tmp/consent-banner-settings.json --force
wp consent-banner export -

Import

wp consent-banner import /tmp/consent-banner-settings.json
wp consent-banner import /tmp/consent-banner-settings.json --replace
wp consent-banner import /tmp/consent-banner-settings.json --dry-run
wp consent-banner import /tmp/consent-banner-settings.json --no-bump-version

Privacy and security notes

Public consent route

The consent submission endpoint is public and rate-limited to reduce abuse.

Admin settings route

Settings endpoints require a valid REST nonce and manage_options capability.

Optional proof log

Consent logs are optional and designed around hashed proof records.
